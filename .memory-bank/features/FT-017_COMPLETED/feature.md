# FT-017: Подключение API для транскрибации (user key)

## What

### Problem

Пользователь загружает видео в браузер, но для видео без субтитров (или для нужного языка) сейчас нет встроенного способа получить текст. Нужно добавить транскрибацию через внешние API (например OpenAI / Groq / OpenRouter), при этом **выбор провайдера и API key задаёт сам пользователь**, а субтитры сохраняются локально в `IndexedDB`.

### Outcome

| Metric ID | Metric | Baseline | Target | Measurement method |
| --- | --- | --- | --- | --- |
| `MET-01` | Возможность получить субтитры без ручной загрузки `.vtt` | Требуется ручной `.vtt` | На странице видео можно сгенерировать `.vtt` через API | Ручная проверка по `SC-01` + сохранённый sample `.vtt` |

### Scope

- `REQ-01` На странице просмотра видео пользователь может выбрать провайдера транскрибации (из поддерживаемого списка) и ввести/вставить API key **локально** (ключ не отправляется на Rails сервер).
- `REQ-02` Пользователь может запустить транскрибацию для загруженного видео и получить результат в виде субтитров `.vtt` **с таймкодами**, пригодный для отображения в `<track kind="subtitles">`.
- `REQ-03` Сгенерированные субтитры сохраняются в `IndexedDB` как `subtitles` для конкретного видео (используя существующую модель хранения видео).
- `REQ-04` Пользователь может скачать сохранённые субтитры и включать/выключать их отображение (используя существующий UI/поведение).
- `REQ-05` Если у видео уже есть субтитры, повторный запуск транскрибации не должен перезаписывать их “молча”: требуется явное действие пользователя для перезаписи.

### Non-Scope

- `NS-01` Не храним API key на сервере и не добавляем бэкенд-прокси для запросов к провайдерам (без новых endpoints).
- `NS-02` Не делаем синхронизацию субтитров между устройствами/браузерами (всё остаётся client-side).
- `NS-03` Не добавляем миграции БД и не меняем `schema.rb`.
- `NS-04` Не реализуем перевод/озвучку в рамках этой delivery-единицы (только транскрибация в текст + `.vtt`).
- `NS-05` Не реализуем client-side экстракцию аудио из видео (например через ffmpeg.wasm) в рамках этого deliverable.
- `NS-06` Не добавляем OpenRouter в первом релизе этой фичи (оставляем как follow-up после стабилизации базового кейса).
- `NS-07` В рамках этой фичи не верифицируем управление языком транскрибации (параметр может быть проброшен, но coverage не является acceptance-критерием).

### Constraints / Assumptions

- `ASM-01` Провайдеры поддерживают получение результата сразу в `.vtt` (или эквивалентном формате с таймкодами) через параметр запроса (например `response_format=vtt`).
- `ASM-02` Текущая страница видео уже умеет сохранять/показывать `.vtt` через `StoredVideoRecord.subtitles?: File` и `setSubtitles(...)` (реализовано ранее, см. `FT-016`).
- `ASM-03` Execution этой фичи ведётся PR-first (ветка + PR до начала code changes), согласно `flows/feature-flow.md`.
- `CON-01` Реализация должна работать в devcontainer окружении и не требовать серверных операций/перезапусков для верификации базовой логики (кроме обычной разработки UI).
- `CON-02` Все артефакты субтитров должны храниться в браузере в `IndexedDB`, совместимо с текущим `StoredVideoRecord.subtitles?: File`.
- `CON-03` У провайдеров есть ограничения на размер загружаемого файла (реализация обязана предвалидировать размер файла и показывать понятное сообщение об ошибке).
- `CON-04` Запрос выполняется напрямую из браузера к внешнему API, поэтому возможны ограничения CORS/сети; UI должен объяснять причину ошибки и что ключ/запрос уходят к провайдеру.

### Decisions

- `DEC-01` Поддерживаемые провайдеры в первом релизе: **OpenAI** и **Groq** (оба через “OpenAI-compatible” HTTP клиент). OpenRouter исключён из `Scope` (см. `NS-06`).
- `DEC-02` Хранение API key: по умолчанию ключ **не сохраняется** и живёт только в памяти страницы; опционально пользователь может включить “Запомнить ключ в этом браузере” (хранение в `localStorage`). IndexedDB для ключей не используем.
- `DEC-03` Поведение при повторной транскрибации: если субтитры уже есть, кнопка “Транскрибировать” требует явного подтверждения “Перезаписать существующие субтитры” (checkbox) перед запуском (см. `REQ-05`, `NEG-03`).

## How

### Solution

Добавить client-side модуль “transcription providers” (OpenAI-compatible) и UI на странице видео: выбор провайдера + API key + параметры (модель/язык при необходимости) + кнопка “Транскрибировать”. По завершении — формировать `.vtt` и сохранять его через существующий `setSubtitles(...)`.

### Change Surface

| Surface | Type | Why it changes |
| --- | --- | --- |
| `video_chat_and_translator/app/frontend/pages/videos/Show.tsx` | code | Добавить UI и запуск транскрибации, сохранение `.vtt` в `IndexedDB` |
| `video_chat_and_translator/app/frontend/features/videos/storage.ts` | code | Использовать существующее хранилище субтитров через `setSubtitles(...)` |
| `video_chat_and_translator/app/frontend/features/videos/transcription/*` | code | Новый модуль провайдер-клиента(ов) и нормализация ошибок/ответов |
| `video_chat_and_translator/app/frontend/features/videos/transcription/__tests__/*` | code | Vitest unit-тесты клиента/обработки ошибок |

### Flow

1. Пользователь открывает страницу видео.
2. Указывает провайдера и API key.
3. Нажимает “Транскрибировать”.
4. Клиент отправляет `File` (видео) в выбранный API.
5. Ответ возвращается как `.vtt` (с таймкодами) и сохраняется в `IndexedDB` как `subtitles`.
6. Пользователь включает субтитры в плеере / скачивает файл.

### Contracts

| Contract ID | Input / Output | Producer / Consumer | Notes |
| --- | --- | --- | --- |
| `CTR-01` | `videoFile: File` → `vttFile: File` | UI (VideosShow) → videos storage | Хранение через существующий `StoredVideoRecord.subtitles?: File` |
| `CTR-02` | `{ provider, apiKey, model?, language? }` → HTTP request | UI → provider client | Ключ хранится локально: in-memory по умолчанию, optional `localStorage` (см. `DEC-02`) |
| `CTR-03` | HTTP response → `.vtt` text | provider client → UI | Требование: `.vtt` включает таймкоды (см. `ASM-01`, `REQ-02`) |

### Failure Modes

- `FM-01` Неверный API key / 401/403 → показать ошибку, не перезаписывать существующие субтитры.
- `FM-02` Провайдер недоступен / CORS / network error → показать ошибку и подсказку, что запрос идёт напрямую из браузера.
- `FM-03` Пользователь запускает транскрибацию повторно при наличии субтитров → без явного подтверждения перезаписи запрос не стартует.
- `FM-04` Ответ провайдера не содержит ожидаемого текста → показать ошибку “не удалось распарсить ответ”.
- `FM-05` Файл превышает лимит провайдера → показать ошибку про размер и рекомендации (например “используйте более короткий файл / дождитесь поддержки извлечения аудио”).

### Negative / Edge Cases

- `NEG-01` Введён неверный API key → запрос возвращает 401/403 → UI показывает ошибку и **не** перезаписывает текущие субтитры.
- `NEG-02` Запрос падает по сети/CORS → UI показывает ошибку с объяснением “запрос выполняется напрямую к провайдеру из браузера”.
- `NEG-03` У видео уже есть субтитры → пользователь нажимает “Транскрибировать” без подтверждения “Перезаписать” → транскрибация не стартует.
- `NEG-04` Провайдер вернул не-`.vtt`/пустой ответ → UI показывает ошибку парсинга, субтитры не меняются.
- `NEG-05` Видео больше лимита (preflight) → UI не отправляет запрос и показывает ошибку про размер (см. `CON-03`, `FM-05`).

## Verify

### Exit Criteria

- `EC-01` На странице видео доступен UI выбора провайдера и ввода API key.
- `EC-02` После успешной транскрибации `.vtt` сохраняется в `IndexedDB` и может быть: (a) скачан, (b) отображён в плеере.
- `EC-03` Ошибки (invalid key / network / oversize / invalid response) отображаются пользователю и не приводят к тихой потере уже сохранённых субтитров.
- `EC-04` Есть минимальный набор unit-тестов (Vitest) для client-side провайдер-клиента и нормализации ошибок.

### Acceptance Scenarios

- `SC-01` Happy path: видео без субтитров → выбрать провайдера → вставить API key → “Транскрибировать” → субтитры `.vtt` сохранены и отображаются в плеере.
- `SC-02` Overwrite guard: видео уже с субтитрами → нажать “Транскрибировать” без подтверждения “Перезаписать” → транскрибация не стартует и субтитры не меняются.
- `SC-03` Error handling: invalid key / network/CORS / oversize / invalid response → UI показывает понятную ошибку и не перезаписывает субтитры.
- `SC-04` Unit tests: провайдер-клиент корректно формирует запрос и нормализует ошибки/ответы (Vitest).

# Research: Загрузка видео и доступ к ранее загруженным видео

**Feature:** 006-video-uploader-page  
**Date:** 06.04.2026  
**Status:** completed

---

## 1. Уточнение задачи

Brief формулирует пользовательский результат, а GitHub Issue #6 добавляет ключевое техническое ограничение:

- страница загрузки нужна для авторизованного пользователя;
- загруженные видео **не должны храниться на сервере**;
- видео должны храниться в **IndexedDB**;
- пользователь должен видеть ранее загруженные видео из IndexedDB;
- после загрузки нужен переход на отдельную страницу с видеоплеером.

Это сразу отсекает серверное хранение через Active Storage как основной вариант реализации.

## 3. Анализ альтернатив

### 3.1 Хранение файлов

#### Вариант A: IndexedDB + нативный браузерный API

Видео и метаданные хранятся напрямую через `indexedDB.open`, `transaction`, `objectStore`.

**Плюсы:**

- полностью соответствует Issue #6;
- не требует серверного хранилища;
- не требует новых зависимостей;
- позволяет хранить `Blob`/`File` и метаданные вместе.

**Минусы:**

- низкоуровневый и многословный API;
- выше вероятность ошибок в транзакциях, миграциях версии БД и обработке исключений;
- хуже читаемость и поддерживаемость TypeScript-кода.

#### Вариант B: IndexedDB + `idb` (рекомендуется)

Использовать тонкую promise-based обёртку над IndexedDB: `idb`.

**Плюсы:**

- полностью соответствует Issue #6;
- остаётся IndexedDB, но без низкоуровневой ceremony;
- хорошо сочетается с TypeScript и небольшим объёмом доменной логики;
- маленькая зависимость, без избыточной абстракции;
- удобно описывать schema versioning и индексы.

**Минусы:**

- это новая npm-зависимость;
- всё равно нужно самостоятельно продумать quota/error flow.

#### Вариант C: IndexedDB + Dexie

Dexie даёт более богатую модель запросов и реактивность.

**Плюсы:**

- сильная ergonomics для IndexedDB;
- удобнее для live-query и нескольких таблиц;
- хорош для offline-first приложений, где модель данных быстро усложняется.

**Минусы:**

- для текущего MVP выглядит тяжелее, чем нужно;
- вводит более крупную абстракцию, хотя сценарий пока простой: загрузка, список, просмотр;
- часть его преимуществ проявится только при появлении поиска, тегов, синхронизации и сложных выборок.

#### Вариант D: Rails Active Storage

Файл уходит на сервер, хранится через Active Storage, список видео приходит с бэкенда.

**Плюсы:**

- хорошо ложится на Rails CRUD;
- проще делать серверные URL, валидации и общую модель `Video`;
- в будущем удобнее синхронизировать историю между устройствами.

**Минусы:**

- **прямо противоречит Issue #6**: видео не должны храниться на сервере;
- потребует миграций и storage tables;
- увеличивает нагрузку на сервер и расход диска;
- расходится с общей продуктовой идеей минимального бэкенда.

**Вывод:** вариант не подходит для этой задачи.

### 3.2 Выбор библиотеки для IndexedDB

| Вариант | Оценка |
|--------|--------|
| Нативный IndexedDB API | Рабочий, но слишком низкоуровневый для повседневной поддержки |
| `idb` | Лучший баланс простоты, размера и контроля |
| Dexie | Хороший вариант, если заранее закладываем сложную offline-first модель |
| localStorage / sessionStorage | Не подходят для видеофайлов по объёму |
| localForage | Упрощает key-value хранение, но хуже подходит для списка, индексов и явной схемы |

**Рекомендация:** `idb`.

### 3.3 Подход к серверным маршрутам

#### Вариант A: Только одна страница `/videos`

Загрузка, список и встроенный плеер на одной странице.

**Плюсы:**

- меньше маршрутов;
- проще серверная часть.

**Минусы:**

- не соответствует issue: после загрузки ожидается переход на отдельную страницу плеера;
- смешивает два разных экрана.

#### Вариант B: `VideosController#index` + `VideosController#show` (рекомендуется)

- `GET /videos` — страница загрузки и списка;
- `GET /videos/:id` — страница плеера.

**Плюсы:**

- соответствует CRUD-only правилу проекта;
- соответствует issue и brief;
- сервер остаётся тонким: он рендерит только shell Inertia-страниц;
- клиентская IndexedDB-логика живёт во frontend-слое.

**Минусы:**

- сервер не знает, существует ли конкретное видео в браузере пользователя;
- страница `show` должна уметь корректно показывать empty-state, если запись не найдена в IndexedDB.

**Рекомендация:** вариант B.

## 4. Анализ узких мест и ограничений

### 4.1 Текущий код не является bottleneck

Серверная часть сейчас очень лёгкая:

- `Pages::DashboardController#show` только рендерит Inertia-страницу;
- shared props содержат только `flash` и `current_user`;
- нет тяжёлых запросов, N+1 и сложной доменной логики вокруг видео.

То есть bottleneck не в текущем Rails-коде. Он смещается в browser-side хранение и работу с файлами.

### 4.2 Главные риски будущей реализации

#### Риск 1: Хранение больших файлов в браузере

Видео могут быстро упереться в quota браузера. IndexedDB подходит для больших данных лучше, чем localStorage, но лимит всё равно конечен и зависит от браузера и устройства.

Следствие:

- нужен явный UX для `QuotaExceededError`;
- нельзя обещать пользователю бесконечное хранилище;
- в private/incognito режимах поведение может быть хуже или временным.

#### Риск 2: Дублирование файла в памяти

Если превращать файл в base64 или держать его долго в React state, память расходуется кратно.

Следствие:

- хранить исходный `File`/`Blob` напрямую в IndexedDB;
- в state держать только метаданные списка;
- на странице плеера создавать `objectURL` лениво и обязательно делать `URL.revokeObjectURL()`.

#### Риск 3: Сервер не может валидировать наличие видео

Маршрут `/videos/:id` сервер отдаст всегда, но конкретного blob может уже не быть в IndexedDB.

Следствие:

- `Videos/Show.tsx` должен уметь обрабатывать состояние "видео не найдено";
- deep-link на страницу плеера не гарантирует существование данных.

#### Риск 4: Данные браузера не равны постоянному облачному хранению

Issue требует хранение в IndexedDB, но это не абсолютная гарантия вечности:

- пользователь может очистить site data;
- браузер может удалить best-effort storage;
- Safari агрессивнее ведёт себя с неиспользуемыми origin-данными.

Следствие:

- в тексте feature/spec нельзя обещать "навсегда";
- нужно обещать доступность "пока данные origin не удалены браузером или пользователем".

#### Риск 5: Нельзя тащить список видео в shared props

Если попытаться смешать клиентское видео-хранилище с Inertia shared props, получится неправильная архитектура:

- shared props будут неполными на сервере;
- каждая страница начнёт нести лишний payload;
- состояние списка станет размазанным между Rails и браузером.

**Решение:** video library должна жить отдельным frontend storage-модулем.

## 5. Обоснованный дизайн решения

### 5.1 Рекомендуемая архитектура

**Backend:**

- добавить `resources :videos, only: [:index, :show]`;
- `VideosController#index` рендерит Inertia-страницу загрузки/списка;
- `VideosController#show` рендерит Inertia-страницу плеера;
- сервер не принимает сам файл и не хранит метаданные видео в БД.

**Frontend:**

- `app/frontend/pages/videos/Index.tsx`:
  - input/drag-and-drop для выбора файла;
  - список ранее загруженных видео из IndexedDB;
  - переход на страницу плеера после успешного сохранения;
- `app/frontend/pages/videos/Show.tsx`:
  - загрузка записи по `id` из IndexedDB;
  - создание `objectURL` для `<video controls>`;
  - fallback UI, если запись не найдена.

**Client storage layer:**

- отдельный модуль, например `app/frontend/features/videos/storage.ts`;
- ответственность:
  - открыть IndexedDB;
  - сохранить видео;
  - получить список видео;
  - получить видео по `id`;
  - удалить видео в будущем;
  - нормализовать ошибки quota / unsupported browser.

### 5.2 Модель данных в IndexedDB

Рекомендуемая база:

- database: `video_chat_and_translator`
- object store: `videos`
- keyPath: `id`

Рекомендуемая структура записи:

```ts
type StoredVideo = {
  id: string
  name: string
  type: string
  size: number
  createdAt: string
  updatedAt: string
  file: Blob
}
```

Опционально можно добавить позже:

- `durationSeconds`
- `posterBlob`
- `lastOpenedAt`

Для MVP это не требуется.

### 5.3 Поток загрузки

```text
1. Пользователь открывает GET /videos
2. React-страница читает список записей из IndexedDB
3. Пользователь выбирает файл через input
4. Frontend валидирует:
   - файл выбран
   - mime/type начинается с video/
5. Storage layer сохраняет Blob + метаданные в IndexedDB
6. После успеха происходит переход на GET /videos/:id
7. Плеер читает запись по id, создаёт object URL и воспроизводит видео
```

### 5.4 Поток списка ранее загруженных видео

```text
1. Index page при mount читает все записи из IndexedDB
2. Список сортируется по createdAt DESC
3. Каждая запись показывает:
   - название файла
   - размер
   - дату загрузки
   - ссылку "Открыть"
4. Клик ведёт на GET /videos/:id
```

### 5.5 Навигация

Нужна новая точка входа для сценария. Самое естественное место:

- добавить ссылку `"Видео"` в `Header.tsx` для авторизованного пользователя;
- опционально добавить CTA на `Dashboard.tsx`.

Это не требует новых архитектурных сущностей и сохраняет сценарий discoverable.

## 6. Что не стоит делать

- Не использовать Active Storage для этой фичи.
- Не сохранять видео в `localStorage`.
- Не кодировать файл в base64 перед сохранением.
- Не делать `POST /videos` на сервер только ради "правильного CRUD", если сервер всё равно не хранит видео.
- Не класть клиентский список видео в Inertia shared props.
- Не обещать кросс-девайсную доступность: задача про конкретный браузер и конкретное устройство.

## 7. Тестовая стратегия для следующего этапа

### Без новых Ruby-гемов

Минимальный обязательный набор:

- request spec на `GET /videos`:
  - гость редиректится по auth-правилам;
  - авторизованный пользователь получает Inertia page;
- request spec на `GET /videos/:id`:
  - гость не допускается;
  - авторизованный пользователь получает Inertia page;
- request spec на наличие навигационной ссылки.

### Для client-side storage логики

Тут текущего RSpec недостаточно, потому что бизнес-логика живёт в браузере. Поэтому для качественной реализации желательно добавить frontend test harness.

Наиболее прагматичный вариант:

- `vitest` + `fake-indexeddb` для unit-тестов storage-модуля.

Если команда хочет избежать новых npm-зависимостей на этом этапе, можно стартовать без них, но тогда ключевая логика IndexedDB останется почти без автоматической проверки. Для этой фичи это заметный риск.

## 8. Итоговая рекомендация

Рекомендуемый путь для `spec` и последующей реализации:

1. Использовать **IndexedDB** как единственное хранилище видео.
2. Для работы с IndexedDB выбрать **`idb`** как минимальную и удобную обёртку.
3. Добавить server-shell маршруты:
   - `GET /videos`
   - `GET /videos/:id`
4. Реализовать отдельный frontend storage-модуль для video library.
5. На index-странице объединить:
   - upload area
   - список ранее загруженных видео
6. На show-странице воспроизводить видео через `URL.createObjectURL(blob)`.
7. Явно обработать состояния:
   - браузер не поддерживает IndexedDB;
   - quota exceeded;
   - видео не найдено по `id`.

Это решение:

- соответствует brief;
- соответствует Issue #6;
- не требует миграций;
- не требует серверного хранения видео;
- оставляет Rails в роли thin shell, что совпадает с текущей архитектурой продукта.

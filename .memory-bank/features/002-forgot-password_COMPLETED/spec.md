# Specification: Система сброса пароля

**Статус:** Completed (04.04.2026)  
**Версия:** 2.0 (grounding после завершения 001-auth-email)  
**Related Issue:** [GitHub Issue #2](https://github.com/OlegPhenomenon/video-chat-and-translator/issues/2)  
**Тех. стек:** Ruby on Rails 8, Inertia.js, React, PostgreSQL, Devise 5.0.3, Redis, Sidekiq 8, RSpec.

## 1. Цель

Дать зарегистрированным пользователям с **подтверждённым email** возможность восстановить доступ к аккаунту при утере пароля. Пользователи **без** подтверждённой почты не должны иметь возможности пройти сценарий восстановления пароля (функция для них недоступна).

### Политика anti-enumeration (зафиксировано)

При запросе сброса пароля **всегда** показывается **одно и то же** пользовательское сообщение об успешном приёме запроса (текст из `i18n`), независимо от того, существует ли пользователь с таким email, подтверждена ли почта или письмо реально ушло в очередь. Это обязательное требование; отдельные сообщения для «email не найден» или «сначала подтвердите почту» **не** показываются на этом шаге.

## 2. Зависимости и контекст

- Фича 001-auth-email **завершена и смержена**. В проекте уже реализованы: Devise (`:database_authenticatable`, `:registerable`, `:confirmable`, `:validatable`), Inertia-страницы входа/регистрации/подтверждения, фоновая отправка писем через Sidekiq, русская локализация UI, AuthLayout, тесты RSpec + FactoryBot.
- Колонки `reset_password_token` (string, unique index) и `reset_password_sent_at` (datetime) **уже присутствуют** в `db/schema.rb` — были созданы в миграции `DeviseCreateUsers` при реализации 001. **Отдельная миграция для 002 НЕ НУЖНА.**
- `config.reset_password_within = 6.hours` **уже задан** в `config/initializers/devise.rb`.
- В спецификации 001 восстановление пароля явно вынесено в out of scope — настоящий документ его закрывает.

## 3. Scope

### Что делаем

- Активация модуля Devise **`:recoverable`** в модели `User` (колонки и конфиг уже есть).
- Политика доступа: письмо со ссылкой сброса уходит **только** если пользователь существует **и** `confirmed_at` заполнен (почта подтверждена). Для несуществующего или неподтверждённого email письмо **не** отправляется; UI при этом следует **политике anti-enumeration** (раздел 1), без обхода подтверждения.
- UI: страница «Забыли пароль?» (ввод email), страница ввода нового пароля по токену; стиль форм согласован с 001 (`AuthLayout`, Tailwind `rounded-md`, форма по центру, `border-gray-300`).
- **Локализация:** весь пользовательский контент страниц сброса пароля хранится в **файлах русской локализации** (`config/locales/ru.yml`) с использованием **i18n**, передаётся через props — **без** захардкоженных русских строк в React. Паттерн — как в `auth/Login` и `auth/Register` из 001.
- Ссылка «Забыли пароль?» на странице входа (`auth/Login`).
- Отправка письма со сбросом **в фоне** (Sidekiq через `deliver_later`), по той же логике, что и письма подтверждения в 001 (метод `send_devise_notification` в `User` уже переопределён).
- **Интеграционные автотесты (RSpec):** покрывают логику запроса сброса, постановку письма в очередь, смену пароля по токену, негативные сценарии. Паттерн — как в `spec/requests/users/sessions_spec.rb` и `registrations_spec.rb`.
- **Доказательства (скриншоты):** каталог `screenshots/002-forgot-password` через **playwright-cli**.

### Что не делаем (Out of Scope)

- OAuth и сторонние identity-провайдеры.
- Смена пароля «в профиле» для авторизованного пользователя (отдельная итерация).
- Кастомные каналы доставки (SMS, push) — только email.

## 5. Инварианты (согласованно с 001)

1. Пароли только в виде хеша (BCrypt через Devise).
2. Сброс пароля не подменяет требование подтверждения email: **неподтверждённый** аккаунт не получает рабочий сценарий восстановления.
3. Один email — один аккаунт.

## 6. Сценарии ошибок и краевые случаи

| Ситуация | Ожидаемое поведение |
| :--- | :--- |
| Email не зарегистрирован | Письмо **не** отправлять. UI: **то же** нейтральное сообщение (**anti-enumeration**). |
| Email зарегистрирован, почта не подтверждена | Reset-письмо **не** отправлять. UI: **то же** сообщение. |
| Невалидный формат email | Клиентская/серверная валидация: ошибка у поля; письмо не отправляется. |
| Новый пароль не проходит правила (**< 6 символов**, `config.password_length = 6..128`) или пароль и подтверждение **не совпадают** | Сообщения у полей (красная рамка + текст), как на Register; пароль в БД **не** обновляется. |
| Истёкший токен сброса (> 6 часов) | Сообщение о недействительной ссылке; предложение запросить сброс снова. |
| Повторный запрос сброса | Новый токен делает старый недействительным (поведение Devise по умолчанию). |
| Ошибка доставки письма (SMTP) | Обработка через очередь (retry Sidekiq); веб-запрос не блокируется. |

---

## 7. Критерии приёмки (сводно)

1. Подтверждённый пользователь может запросить сброс, получить письмо, задать новый пароль и войти.
2. Неподтверждённый пользователь **не** может завершить восстановление.
3. UI визуально согласован с формами авторизации из 001 (`AuthLayout`, Tailwind); все тексты через i18n.
4. Письма уходят через фоновую обработку (Sidekiq).
5. Интеграционные автотесты (RSpec) зелёные.
6. Пути и имена маршрутов сверены с актуальным `routes.rb`.
7. Скриншоты в `screenshots/002-forgot-password` через playwright-cli.
8. Работа не считается завершённой, пока не выполнены все требования настоящей спецификации.

## 8. Ограничения на реализацию

1. **Аутентификация:** только Devise `:recoverable`.
2. **Очередь:** только через `deliver_later` + Sidekiq (уже настроено).
3. **Схема БД:** миграция **НЕ нужна** — колонки уже есть. Если вдруг потребуются изменения — согласовать с владельцем.
4. **UI:** Tailwind CSS, `AuthLayout`, `useForm` из Inertia; текст через i18n props.
5. **Маршруты:** только стандартные CRUD-паттерны; кастомные действия через namespaced контроллеры.
6. **Anti-enumeration:** на шаге ввода email — без исключений.

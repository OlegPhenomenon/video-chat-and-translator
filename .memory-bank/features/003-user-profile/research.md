### Текущая конфигурация Devise (важные параметры)

```ruby
# config/initializers/devise.rb
config.reconfirmable = false          # email меняется мгновенно, без повторного подтверждения
config.password_length = 6..128
config.reset_password_within = 6.hours
config.confirm_within = 24.hours
```

**Критический момент:** `reconfirmable = false` означает, что при смене email новый адрес вступает в силу немедленно без подтверждения. Это потенциальная проблема безопасности.

## 2. Анализ альтернатив

### 2.1 Подход к контроллеру

#### Вариант A: Использовать встроенный Devise `RegistrationsController#update`

Devise из коробки предоставляет `edit` и `update` действия в `RegistrationsController`, которые работают через метод `update_with_password`.

**Плюсы:**
- Минимум кода — уже реализовано в Devise
- Встроенная логика `update_with_password`: требует `current_password` только при смене пароля
- Маршруты уже сгенерированы (`edit_user_registration_path`, `PUT /users`)

**Минусы:**
- Нарушает архитектурное правило проекта: контроллеры должны содержать только CRUD
- Смешивает логику регистрации и управления профилем в одном контроллере
- Сложнее кастомизировать (email и пароль в одной форме)
- `RegistrationsController` уже кастомизирован для регистрации — добавление профиля усложнит

#### Вариант B: Отдельные контроллеры в namespace `Users::Profile` (рекомендуется)

Создать отдельные контроллеры по архитектурным правилам проекта:
- `Users::ProfileController` — показ профиля (CRUD: `show`)
- `Users::Profile::EmailsController` — смена email (CRUD: `update`)
- `Users::Profile::PasswordsController` — смена пароля (CRUD: `update`)

**Плюсы:**
- Соответствует архитектуре проекта (CRUD-only контроллеры, namespaced для специфических действий)
- Чёткое разделение ответственности
- Легко тестировать каждый контроллер отдельно
- Можно использовать разные политики валидации для email и пароля

**Минусы:**
- Больше файлов (3 контроллера вместо 1)
- Нужно вручную реализовать логику, которую Devise предоставляет из коробки

#### Вариант C: Один `Users::ProfileController` с concern-ами

Один контроллер `Users::ProfileController` с `show` и `update`, но логику обновления email и пароля вынести в concern-ы.

**Минусы:**
- `update` одного контроллера обрабатывает две разные сущности — нарушает SRP
- Concern-ы для контроллеров оправданы для общей логики, не для разделения бизнес-логики

**Рекомендация: Вариант B** — отдельные контроллеры. Это соответствует архитектурным правилам проекта и обеспечивает чистое разделение.

### 2.2 Подход к смене email

#### Вариант A: Мгновенная смена (текущее поведение, `reconfirmable = false`)

Email меняется сразу после сохранения. Подтверждение не требуется.

**Плюсы:**
- Простота реализации
- Минимальный UX friction

**Минусы:**
- **Риск безопасности**: пользователь может потерять доступ при опечатке в email
- **Риск безопасности**: при компрометации сессии злоумышленник меняет email мгновенно
- Не соответствует best practices

#### Вариант B: Включить `reconfirmable = true` (рекомендуется)

При смене email:
1. Новый email сохраняется в `unconfirmed_email`
2. Отправляется письмо подтверждения на **новый** адрес
3. До подтверждения пользователь продолжает использовать старый email
4. После клика по ссылке — `unconfirmed_email` переносится в `email`

**Плюсы:**
- **Безопасно**: защита от опечаток и компрометации сессии
- Колонка `unconfirmed_email` уже существует в schema
- Devise полностью поддерживает этот flow из коробки
- Стандартная практика в индустрии

**Минусы:**
- Сложнее UX: нужно показывать статус "ожидает подтверждения"
- Нужно письмо подтверждения (уже реализовано для регистрации)

#### Вариант C: Требовать текущий пароль для смены email

Дополнительная мера — при смене email запрашивать текущий пароль.

**Оценка:** Можно комбинировать с Вариантом B для максимальной безопасности, но увеличивает friction. На данном этапе достаточно reconfirmable.

**Рекомендация: Вариант B** — включить `reconfirmable = true`. Инфраструктура уже готова (колонка в БД, email-шаблоны, Sidekiq).

### 2.3 Подход к смене пароля

#### Вариант A: Devise `update_with_password` (рекомендуется)

Использовать встроенный метод модели `user.update_with_password(params)`:
- Требует `current_password`, `password`, `password_confirmation`
- Валидирует текущий пароль перед сменой
- Автоматически хеширует новый пароль

**Плюсы:**
- Проверенная, безопасная реализация
- Уже часть Devise — не нужны дополнительные гемы
- BCrypt валидация из коробки

**Минусы:**
- Нет (это стандартный подход)

#### Вариант B: Кастомная реализация

Написать свою логику проверки текущего пароля и обновления.

**Минусы:**
- Дублирование того, что Devise уже делает
- Риск ошибок безопасности
- Нарушение DRY

**Рекомендация: Вариант A** — использовать `update_with_password`. Нет причин не использовать встроенный метод.

### 2.4 Подход к фронтенду

#### Вариант A: Единая страница профиля с двумя секциями

Одна страница `Profile/Show.tsx` с двумя формами:
- Секция "Email" (текущий email, поле для нового)
- Секция "Пароль" (текущий пароль, новый, подтверждение)

**Плюсы:**
- Всё в одном месте — удобно для пользователя
- Меньше навигации
- Стандартный паттерн (GitHub, GitLab, Notion используют этот подход)

**Минусы:**
- Две формы на одной странице — нужно аккуратно обрабатывать ошибки

#### Вариант B: Отдельные страницы для email и пароля

Три страницы: `Profile/Show.tsx`, `Profile/EditEmail.tsx`, `Profile/EditPassword.tsx`.

**Минусы:**
- Излишне для двух полей
- Больше навигации для пользователя

**Рекомендация: Вариант A** — единая страница с двумя независимыми формами. Каждая форма отправляет данные на свой endpoint.

## 3. Анализ зависимостей и узких мест

### 3.1 Необходимые изменения (без новых гемов)

Все необходимые инструменты уже есть в проекте:

| Задача | Инструмент | Статус |
|--------|-----------|--------|
| Хеширование паролей | Devise (BCrypt) | Есть |
| Валидация email | Devise `:validatable` | Есть |
| Повторное подтверждение email | Devise `:confirmable` + `reconfirmable` | Нужно включить |
| Email-шаблон подтверждения | `confirmation_instructions.html.erb` | Есть (от 001-auth-email) |
| Фоновая отправка email | Sidekiq + `send_devise_notification` | Есть |
| Форма с валидацией | Inertia `useForm` | Есть |
| Уведомления | Toast компонент | Есть |
| Стилизация | Tailwind CSS | Есть |

### 3.2 Потенциальные узкие места

1. **Сессия после смены пароля:** Devise автоматически разлогинивает пользователя при смене пароля (invalidation by password salt). Нужно вызвать `bypass_sign_in(user)` после успешной смены, чтобы пользователь оставался залогиненным.

2. **`reconfirmable` и существующие пользователи:** Включение `reconfirmable = true` не влияет на текущих подтверждённых пользователей. Влияет только на будущие изменения email.

3. **CSRF-токен при Inertia:** Inertia.js автоматически обрабатывает CSRF — проблем не ожидается.

4. **Навигация к профилю:** Сейчас Landing.tsx содержит только кнопку выхода. Нужно добавить ссылку на профиль.

## 4. Обоснованный дизайн решения

### Архитектура

```
Routes:
  GET    /users/profile                → Users::ProfileController#show
  PATCH  /users/profile/email          → Users::Profile::EmailsController#update
  PATCH  /users/profile/password       → Users::Profile::PasswordsController#update

Controllers:
  app/controllers/users/profile_controller.rb          # show — отображение профиля
  app/controllers/users/profile/emails_controller.rb   # update — смена email
  app/controllers/users/profile/passwords_controller.rb # update — смена пароля

Frontend:
  app/frontend/pages/profile/Show.tsx                  # Единая страница профиля

Config changes:
  config/initializers/devise.rb                        # reconfirmable = true
```

### Поток смены email

```
1. Пользователь видит текущий email на странице профиля
2. Вводит новый email → PATCH /users/profile/email
3. Backend: user.update(email: new_email)
   - Devise сохраняет новый email в unconfirmed_email
   - Отправляет письмо подтверждения на новый адрес
4. Frontend: показывает уведомление "Письмо подтверждения отправлено на new@email.com"
5. На странице профиля отображается: "Текущий: old@email.com | Ожидает подтверждения: new@email.com"
6. Пользователь кликает ссылку в письме → email обновляется
```

### Поток смены пароля

```
1. Пользователь на странице профиля заполняет:
   - Текущий пароль
   - Новый пароль
   - Подтверждение нового пароля
2. PATCH /users/profile/password
3. Backend: user.update_with_password(params)
   - Валидирует текущий пароль
   - Обновляет пароль
   - bypass_sign_in(user) — сохраняет сессию
4. Frontend: показывает уведомление "Пароль успешно изменён"
```

### UI-макет страницы профиля

### Ключевые решения

| Решение | Обоснование |
|---------|-------------|
| Отдельные контроллеры для email и пароля | Архитектурные правила проекта (CRUD-only, namespaced) |
| `reconfirmable = true` | Безопасность: защита от опечаток и компрометации; инфраструктура уже готова |
| Единая страница профиля | UX: минимум навигации для двух простых операций |
| `update_with_password` для пароля | Проверенный метод Devise, без дополнительных гемов |
| Не требовать `current_password` для email | Devise по умолчанию не требует; reconfirmable обеспечивает безопасность |
| Без новых гемов | Все необходимые инструменты уже в проекте |

- Аватар / фото профиля — нет в brief

- Удаление аккаунта — отдельная фича
- 2FA — отдельная фича

## 5. Риски и митигации

| Риск | Вероятность | Митигация |
|------|------------|-----------|
| Пользователь вводит несуществующий email при смене | Средняя | `reconfirmable` не даст потерять доступ — старый email остаётся активным |
| Письмо подтверждения попадает в спам | Низкая | Уже решено для регистрации (Sidekiq + настроенный mailer) |
| Пользователь забыл текущий пароль при смене | Средняя | На странице профиля ссылка на "Забыли пароль?" (фича 002) |
| Конфликт маршрутов с Devise passwords | Низкая | Namespace `profile/passwords` не пересекается с `users/passwords` |

## 6. Тестовая стратегия

### Request specs (RSpec)

**`spec/requests/users/profile_spec.rb`:**
- Неавторизованный доступ → редирект на логин
- Отображение текущего email

**`spec/requests/users/profile/emails_spec.rb`:**
- Успешная смена email → `unconfirmed_email` обновлён
- Невалидный email → ошибка валидации
- Пустой email → ошибка
- Email уже занят → ошибка
- Тот же email → ошибка или no-op

**`spec/requests/users/profile/passwords_spec.rb`:**
- Успешная смена пароля
- Неверный текущий пароль → ошибка
- Пароли не совпадают → ошибка
- Слишком короткий пароль → ошибка
- Пользователь остаётся залогинен после смены

## 7. Итог

Для реализации фичи "Профиль пользователя" **не требуются новые гемы или библиотеки**. Всё необходимое уже есть в проекте. Основные технические решения:

1. **Архитектура:** 3 контроллера в namespace `Users::Profile` (show, email update, password update)
2. **Безопасность:** Включить `reconfirmable = true` для безопасной смены email
3. **Фронтенд:** Единая страница профиля с двумя независимыми формами
4. **Пароль:** Использовать Devise `update_with_password` — требует текущий пароль
5. **Email:** Devise reconfirmable flow — письмо подтверждения на новый адрес

Решение готово к этапу Spec.

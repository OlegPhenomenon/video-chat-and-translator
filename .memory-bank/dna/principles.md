# Principles

1. **SSoT (Single Source of Truth).** Каждый технический или бизнес факт имеет ровно одного каноничного владельца (canonical owner). Дублирование информации = дефект.
2. **Атомарность.** Один файл = одна конкретная тема. Если файл разрастается, он должен быть декомпозирован.
3. **Компактность.** Документ должен оставаться лаконичным и оптимизированным для чтения как разработчиками-людьми, так и ИИ-агентами (Claude / Cursor и т.д.).
4. **Progressive disclosure.** Сначала высокоуровневый обзор, затем ссылки вглубь на детали реализации. Чтение и погружение всегда сверху вниз.
5. **WHY / WHAT / HOW.** Раздел `adr/` отвечает на вопрос "почему", `features/` и спецификации отвечают на вопрос "что", а сам `код` является единственным источником правды для ответа на вопрос "как".
6. **Code vs Docs.** Код владеет фактической реализацией деталей. Документация Memory Bank фиксирует намерения, архитектурные контракты и обоснования. Не нужно пересказывать код в документации.
7. **Index-first.** Каждый документ должен присутствовать в навигационном индексе. "Сиротский" (orphan) файл считается дефектом.
8. **Аннотированные ссылки.** Любая Markdown-ссылка должна объяснять контекст: что находится по ссылке и зачем читателю туда переходить.
9. **ADR (Architecture Decision Records).** Каждое крупное решение (например, выбор стека Ruby on Rails + Inertia, вынесение ML в Web Workers WASM) обязательно фиксируется отдельным документом в разделе ADR.
10. **Agentic Maintenance.** ИИ-Агенты (наравне с людьми) являются полноценными мейнтейнерами проекта и **обязаны** синхронизировать Memory Bank при рефакторингах и добавлении новых фич, ссылаясь на корневой индекс перед началом каждой задачи.

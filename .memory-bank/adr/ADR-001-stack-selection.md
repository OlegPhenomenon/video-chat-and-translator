# ADR-001: Core Architecture Stack (Rails + Inertia + Client ML)

## Context
Проекту требуется надежный слой управления бизнес-логикой (аккаунты, интервальные повторения Anki, интеграция с Telegram-ботом). Однако главная ценность продукта — видеоперевод и ИИ-тьютор — подразумевает тяжелые медиа-вычисления. Традиционный подход, где бэкенд обрабатывает аудио/видео, недопустимо дорог и не масштабируется. 

Кроме того, стандартное разделение SPA (Single Page Application) и REST API вводит излишний overhead на поддержку роутера и сериализации на клиенте: нам нужен монолитный, но реактивный подход.

## Decision
Мы принимаем следующую комбинированную архитектуру:
1. **Ruby on Rails 8** в качестве единственного источника серверной истины (Postgres, ActiveJob, Telegram-боты).
2. **Inertia.js** в качестве транспортного слоя маршрутизации. Rails контроллеры будут напрямую отдавать состояния на клиент без необходимости создавать GraphQL / REST.
3. **React** (совмещенный с Vite/Tailwind) как UI-фреймворк.
4. **Client-side ML (ONNX WASM / WebGPU)** вынесенный в **Web Workers**. Все аудио/видео вычисления происходят исключительно на ресурсах пользователя в браузере.

## Consequences
- **Снижение затрат:** Бэкенд-инфраструктуре не нужны инстансы с GPU, достаточно дешевых CPU контейнеров.
- **Приватность:** Видеопотоки пользователя не покидают его устройство.
- **Производительность разработки:** Inertia + Rails позволяет отказаться от сложного клиентского стейт-менеджмента для CRUD операций.
- **Зависимость от клиентских девайсов:** Приложение может падать на слабых устройствах (Out of Memory). Необходимо закладывать fallback UX, где модель транскрибации не может быть загружена.

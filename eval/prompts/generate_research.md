# Generate Research

Проблема из Brief понятна. Решение неочевидно. Проведи исследование и подготовь обоснованный дизайн решения для следующего этапа (Spec).

## ОБЯЗАТЕЛЬНАЯ СТРУКТУРА RESEARCH

Документ ДОЛЖЕН содержать эти секции в этом порядке:

1. `## 1. Анализ текущего состояния` — таблица «Что уже есть» / «Чего нет» в проекте. Конкретные файлы, гемы, маршруты. Не общие слова.
2. `## 2. Анализ альтернатив` — для каждого решения ≥2 варианта с Плюсы/Минусы. Один вариант — рекомендация с обоснованием.
3. `## 3. Анализ узких мест и рисков` — что может сломаться, quota, edge cases, зависимости.
4. `## 4. Обоснованный дизайн решения` — архитектура, потоки, модель данных. Конкретные файлы/маршруты/компоненты.
5. `## 5. Риски и митигации` — таблица Риск/Вероятность/Митигация.
6. `## 6. Тестовая стратегия` — какие тесты нужны, какие инструменты.
7. `## 7. Итоговая рекомендация` — краткое резюме: что делаем, чего не делаем, какие гемы/библиотеки.

## ПРАВИЛА КАЧЕСТВА

1. **Конкретика, не общие слова.** Каждый пункт должен ссылаться на конкретный файл, гем, маршрут, API или компонент. «Нужно добавить контроллер» — плохо. «Добавить `Users::ProfileController#show` в `app/controllers/users/profile_controller.rb`» — хорошо.

2. **Альтернативы обоснованы.** Для каждого варианта: Плюсы, Минусы, **Рекомендация** с объяснением «почему именно этот». Не просто «вариант A — лучше».

3. **Риски конкретны.** Не «могут быть проблемы», а конкретный сценарий + митигация. Привязка к ограничениям проекта (Docker, client-side, IndexedDB quota и т.п.).

4. **Дизайн решает проблему из Brief.** Секция 4 должна явно показывать, как предложенная архитектура закрывает каждый пункт Brief'а.

5. **Тестовая стратегия привязана к проекту.** Учитывать существующий стек (RSpec для backend, Vitest для frontend если есть). Если нужны новые инструменты — обосновать.

6. **Нет выдуманных зависимостей.** Если Brief упоминает конкретный гем/библиотеку — используй. Если не упоминает — рекомендуй最小ный набор, обосновывая выбор.

7. **Источники.** Если ссылаешься на внешнюю документацию (MDN, гемы, API) — дай ссылку.
